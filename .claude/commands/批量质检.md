# 角色

你是**质检 AI**。任务：对 `$ARGUMENTS` 指向的数据目录进行深度质检。**所有输出必须全程使用中文。**

质检不是重新评分，而是验证评分文件的合规性、一致性和真实性。

一条数据 = 同一任务的 4 个文件：
- qwen 评分 `*.quality.toml`、claude 评分 `*.quality.toml`
- qwen 轨迹 `*.jsonl`、claude 轨迹 `*.jsonl`

---

# 第 0 步：定位目录、枚举任务、随机抽样

1. 解析 `$ARGUMENTS` 为数据目录。该目录应含 `scores/qwen`、`scores/claude`、`trajectories/qwen`、`trajectories/claude` 四个子目录（scores 目录兼容 `scores`/`Scores`/`score`/`Score`/`secore` 拼写）。
2. 用 Glob 列出全部 `*.quality.toml` 与 `*.jsonl`。
3. **按编号后缀聚合任务**，兼容两种命名：
   - 新命名 `{model}-{编号}`（如 `qwen-0001.quality.toml`）
   - 旧命名 `CT-{编号}`（如 `CT-0001.quality.toml`）
4. 对每个编号判**文件完整性**：4 处是否各有对应文件。
5. **随机抽样**：从全部任务中随机抽取**最多 10 个**任务 ID 作为本次抽检对象。若总任务数 ≤ 10 则全部检查。
6. 整理抽检清单：`[(编号, claude_toml, qwen_toml, claude_jsonl, qwen_jsonl), ...]`，记录每个文件的**绝对路径**。

# 8 项质检标准

## 检查项 1：TOML 合法性
- TOML 是否可解析
- criterion 数量是否在 **5-15** 之间
- 每个 criterion 是否包含全部 7 个必需字段：`name`、`description`、`type`、`points`、`weight`、`score`、`rationale`
- `type` 固定为 `"likert"`；`points` 固定为 `5`；`score` 为整数 1-5；`weight` 为正数

## 检查项 2：维度唯一性（name）
- name 格式是否合法 snake_case（仅小写字母、数字、下划线）
- 是否存在泛名或序号名（如 `criterion_1`、`quality_score`、`general_quality`）
- 同一文件内 name 是否有重复

## 检查项 3：维度原子性（description）
- 是否以「该维度评价」或「该维度衡量」开头
- 是否包含完整的 1-5 分各档位定义
- 是否原子化（只评一个主要质量点）
- 是否为中文且与任务相关

## 检查项 4：定制维度检查（description 定制性）
- description 是否结合了本次任务的实际内容定制生成
- description 是否与当前任务的 JSONL 轨迹相关（不能是与任务无关的通用模板）
- 不同任务的 description 不应完全一致

## 检查项 5：事实篡改检查（rationale）——**最重要**
- rationale 是否为中文
- 是否包含具体轨迹证据
- 是否与 description 档位定义一致
- 是否有空泛模板化表述
- **重点：对照 JSONL 轨迹文件，逐条验证 rationale 中引用的证据是否真实存在**
- 各条 rationale 是否相互雷同

**严重程度区分（仅严重篡改算 FAIL）：**
- **严重篡改（FAIL）**：完全编造不存在的文件名、命令、测试结果、工具调用；将失败说成成功或成功说成失败；捏造用户反馈或验证结果
- **轻微不准确（不算篡改，PASS）**：文件名/路径细节略有出入但指向正确文件；命令参数记忆偏差但操作方向正确；对证据的概括性描述与原文措辞不完全一致但含义正确

## 检查项 6：用户提问一致性（成对检查）
- 同一任务的 claude 和 qwen 轨迹文件首轮用户提问
- 任务意图、问题对象和主要约束是否一致（不要求逐字相同）
- 不一致则标记为严重问题

## 检查项 7：成对评分框架一致性
- name/description/type/points/weight 是否逐字一致
- 只有 score 和 rationale 可以不同
- criterion 顺序和数量是否一致

## 检查项 8：Passrate 与 Gate 检查
- passrate 计算：`sum(score × weight) / sum(points × weight)`
- Gate 三条件**必须全部满足**才 PASS：
  1. qwen passrate < 0.7
  2. claude passrate > 0.7
  3. claude passrate - qwen passrate > 0.2
- 任意一个不满足即为 Gate FAIL，判定时必须逐条列出每条结果

# 执行方式：并行子 agent（每个任务一个）

1. **在同一条消息里**用 Agent 工具并行启动 N 个子 agent（N = 抽检任务数），每个任务一个子 agent，互不依赖，**必须并行**。
2. 每个子 agent 的 prompt 必须包含：
   - 该任务的 4 个文件的**绝对路径**
   - 上面 8 项质检标准的**完整内容**
   - 明确指令：先读取评分文件（TOML），再读取轨迹文件（JSONL），按检查项 1-8 逐项检查
   - 事实篡改检查时必须对照轨迹文件中的实际内容验证 rationale 中的引用
   - 要求返回每个检查项的 PASS/FAIL 及具体证据
3. 子 agent **只做核验与返回结论，不写文件**。
4. 解析 JSONL：逐行 JSON；第一条真实 user query 取首个 `role=="user"` 且含真实文本的消息（跳过系统/工具回灌）。
5. Windows 上用 Python 打印中文先 `sys.stdout.reconfigure(encoding='utf-8')`。

# 汇总与产出（主 agent）

收齐所有子 agent 结果后，**严格按以下固定格式输出质检报告**，不得改变结构。所有内容使用中文。

**同时**将报告写入 `$ARGUMENTS` 指向的数据目录下，文件名为 `批量质检报告.md`。

# 注意

- 必须全程以中文回答，所有输出内容均使用中文。
- **必须使用 Agent 工具并行启动子 agent，所有 Agent 调用必须在同一条消息中发出（并行执行），严禁串行逐个启动。**
- 每个子 agent 分析一个任务，子 agent 的 prompt 必须包含完整的文件路径和检查要求。
- **事实篡改检查（检查项 5）是最重要的检查项**，必须对照轨迹文件逐条验证。
- 输出必须严格遵循上面的固定格式，不得增减章节或改变结构。
- 所有检查项都要给出具体证据和说明，不能只写 PASS/FAIL。
- 如果某个文件缺失，在对应检查项标注「文件缺失」并跳过成对检查。
- passrate 用 `sum(score × weight) / sum(points × weight)` 计算，分母按文件实际权重动态计算。
- 只做**核验与判定**，**绝不修改**被检查的数据文件。
- 子 agent 只返回结论，**写文件由主 agent 统一完成**，避免并发写冲突。
