# quality.toml 生成规范与评分标准

## 0. AI 使用方式

你是评分 AI。阅读本文档后，按以下顺序执行：

1. 收集评分证据时先做分层取证：先读任务目标、最终回复、窗口 transcript、人工备注、acceptance criteria、工具调用索引、文件变更摘要和失败摘要，不要默认逐行展开完整 JSONL。
2. 基于快速索引判断是否命中 Bad Pattern；如命中，只需在评分说明中额外指出，不生成文件，也不写入 `quality.toml`。
3. 从本文 20 个候选 criterion 中，依据真实轨迹证据选择 7-10 个最适合本次轨迹的维度。
4. 只在维度评分或 Bad Pattern 判断需要更细证据时，回到原始 JSONL 定点读取相关片段。
5. 如果是 qwen/claude 成对比较，同一任务对必须先生成同一份共享评分框架，保证 criterion 顺序、`name`、`description`、`type`、`points`、`weight` 完全一致，只有 `score` 和 `rationale` 可以不同。
6. 再根据证据给选中的 criterion 打 1-5 分。
7. 写 `quality.toml` 时只输出合法 TOML，只包含选中的 7-10 个 `[[criterion]]`，不得附加 Bad Pattern、Markdown、标题或解释。
8. 所有字符串字段都必须使用普通双引号 `"..."`，不得使用 TOML 多行字符串。
9. `rationale` 写中文证据和扣分原因，不要写“命中 Bad Pattern: xxx”这类标签化结论。
10. 输出前检查 TOML 可解析、字段完整、分数范围正确、维度数量正确、4/5 分符合高分校准与封顶规则、Bad Pattern 未进入 TOML。

如果系统要求同时给出 TOML 和 Bad Pattern 命中说明，先输出纯 `quality.toml`；Bad Pattern 命中说明只作为评分说明中的额外文字指出，不创建单独文件。

## 1. 目标

`quality.toml` 用于评价一次 coding-assistant 执行轨迹的质量。评分对象不是单个代码文件，而是一次模型完成任务的完整过程，包括用户任务、对话记录、JSONL 轨迹、窗口 transcript、人工备注和评判用验收标准。

评分文件不再固定使用 7 个维度。评分 AI 必须从 20 个候选维度中选择 7-10 个最能反映当前 JSONL 轨迹质量差异的维度。选择维度时应基于真实证据，而不是机械套用固定模板。

`description` 不再使用固定模板原文。评分 AI 必须结合本次任务、JSONL 轨迹、验收标准和选中的 criterion，自主生成当前任务专属的中文单行评分标准。格式必须大致一致：先说明该维度评价什么，再写清 1分、2分、3分、4分、5分各档位标准。

评分时必须识别 Bad Pattern：任务偷懒、缺少澄清、该搜索不搜索、环境误判、忽略项目规则、附件处理不足、只做准备动作、错误使用工具/协议、平台误判、并行整合不足等。

Bad Pattern 命中结果必须额外指出，但不得写入 `quality.toml`。`quality.toml` 只能包含选中的 `[[criterion]]`；Bad Pattern 命中说明只作为评分说明中的额外文字指出，不生成单独文件。

## 2. 输入证据

生成评分时应综合以下证据：

1. 原始对话或任务备注。
2. 压缩后的 JSONL trajectory transcript。
3. 人工备注。
4. TOML 评分模板或当前候选维度池。
5. 固定工作流中的窗口 transcript。
6. 仅供评分员使用的 acceptance criteria。
7. 项目内约束文件或环境说明，例如 `AGENTS.md`、`CLAUDE.md`、README、TESTING_GUIDE、skill、tool schema、lockfile、Dockerfile、environment 文件。
8. 用户提供或 workspace 中存在的附件、日志、截图、PDF、zip、xmind 等材料。
9. 当任务明确依赖外部时，相关 GitHub issue、PR、release note 或官方文档证据。

要求：

- 如果同一模型配置包含多个问题，应给出该模型的一组总体评分，而不是逐题输出多个 TOML。
- JSONL 轨迹和可见窗口 transcript 都要参考；如果二者冲突，应在中文理由中说明不确定性。
- 应检查 JSONL 中的 `tool_use`、`tool_result`、`is_error`、stderr/stdout、超时、权限拒绝、文件不存在、参数错误等证据，识别工具调用失败及其影响。
- 应区分 `planning_action` 和 `substantive_action`：加载 skill、写 todo、复述需求、声明计划不算实质进展；读取关键文件、定位错误、解析附件、调用正确 tool、修改代码、生成文件、运行验证才算实质进展。
- 原始执行 prompt 通常不会包含 acceptance criteria；acceptance criteria 只作为评分证据，不应反推出模型当时应当知道的隐藏信息。
- 如果 acceptance criteria 里包含 Implicit 或 Penalty，应据此判断完成质量和坏行为，但不要要求模型必须逐字知道隐藏条款。
- 如果发现 Bad Pattern，应在评分说明中额外指出，再把对应影响折算到选中的评分维度中。TOML 的 `rationale` 可以描述具体证据和扣分原因，但不要写“命中 Bad Pattern: xxx”这类标签化结论。

## 2.1 JSONL 读取效率与分层取证

评分 AI 的目标不是复原完整对话，而是为选中的评分维度找到足够、具体、可核查的证据。不得因为不确定就直接全量展开原始 JSONL；应先压缩索引，再按需回读。

推荐读取顺序：

1. 快速索引：先提取原始任务、用户补充、最终回复、工具调用列表、`tool_result.is_error`、stderr/stdout 摘要、文件创建或修改记录、测试/构建/lint 命令、权限拒绝、超时、附件处理记录和人工备注。
2. 轨迹摘要：基于快速索引生成内部用的 trajectory 摘要，至少包含任务目标、主要操作路径、关键工具调用、失败恢复、最终交付、验证证据、可能命中的 Bad Pattern 和证据缺口。
3. 维度初选：先根据摘要从 20 个候选维度中选择 7-10 个，不要为了填维度去读无关片段。
4. 按维度补证：每个选中维度优先回读 1-3 个最相关片段；只有证据冲突、最终自述可疑、工具失败影响不明或 Bad Pattern 难以确认时，才扩大读取范围。
5. 成对比较：qwen 与 claude 应先分别生成快速索引，再基于两个索引共同生成同一份共享评分框架；不要因为某一边 JSONL 更长就选择更多维度或给出更笼统的理由。

优先回读的 JSONL 片段包括：

- 用户多轮纠正、明确补充约束或表达不满意的上下文。
- 失败 `tool_result` 前后，包括错误参数、权限拒绝、文件不存在、超时、测试失败和 fallback 动作。
- 文件写入、补丁应用、生成附件、解析附件、运行测试、构建、lint、dry-run 的上下文。
- 最终回复前的关键操作，尤其是最终声称已完成、已验证或已修复的内容。
- 可能命中 Bad Pattern 的片段，例如只做计划、转嫁给用户、忽略附件、该搜索不搜索、环境误判、工具协议误用。

如果摘要证据已经足够支持某个维度评分，不要继续展开无关 JSONL。`rationale` 应写压缩后的具体证据和判断，例如“连续两次命令失败后没有 fallback”“只运行 lint 未覆盖测试”“最终回复声称完成但轨迹中没有文件修改证据”，而不是复述完整对话。

## 3. 维度选择规则

每次评分必须从 20 个候选 criterion 中选择 7-10 个维度。选择依据是 JSONL 轨迹中真实出现的任务特征和行为证据。

选择流程：

1. 先读任务目标和 acceptance criteria，判断本次任务是修复、功能、测试、文档、构建、解释、附件处理、工具调用、环境诊断、并行工作流还是安全合规。
2. 再基于 JSONL 快速索引统计模型实际做了什么：读了哪些文件、调用了哪些工具、是否失败、是否修复、是否验证、是否只计划、是否转嫁给用户。
3. 选择 7-10 个最能区分本次轨迹质量的维度。
4. 如果某个维度没有任何可观察证据，不要选择它。
5. 如果证据很多，优先选择与核心目标、失败点、Bad Pattern、验收标准和最终可用性最相关的维度。
6. 如果证据不足但仍必须评分，选择最基础的维度，并在 `rationale` 中写明“未看到相关证据”。

成对评分规则：

- 对同一任务的 qwen 与 claude 轨迹，应先基于任务和两边 JSONL 共同选择一组 7-10 个 criterion。
- 两个模型的 `quality.toml` 必须使用同一份共享评分框架，避免因为维度、权重或 description 不同导致 passrate 不可比。
- 共享评分框架包括 criterion 顺序、`name`、`description`、`type`、`points`、`weight`。同一个问题下，qwen 与 claude 的这些字段必须逐字一致。
- `description` 虽然由评分 AI 结合真实 JSONL 自主生成，但在成对比较中只能生成一次，并同时用于 qwen 与 claude；不得分别为两个模型生成不同的评分标准。
- 两个模型的 `quality.toml` 中只允许 `score` 和 `rationale` 不同；`rationale` 应分别结合各自轨迹证据写作。
- 如果某个维度只在一边有证据，另一边仍按同一维度评分，并在理由中说明缺少证据或未执行相关动作。

常见选择建议：

- 任何代码改动任务通常应考虑 `semantic_understanding_and_logical_reasoning`、`engineering_quality_and_completeness`、`delivery_completeness_and_usability`。
- 工具调用多、失败多或需要恢复时，选择 `tool_usage_and_failure_recovery`。
- 测试、构建、lint、dry-run 是关键证据时，选择 `testing_and_verification_rigor`。
- 轨迹存在最终自述与实际证据不一致时，选择 `evidence_grounding_and_trace_fidelity`。
- 任务模糊、用户多轮补充或模型需要提问时，选择 `requirements_clarification_and_scope_control`。
- 涉及附件、PDF、图片、zip、日志包时，选择 `attachment_and_artifact_handling`。
- 涉及外部 issue、PR、官方文档或搜索时，选择 `external_research_and_source_use`。
- 涉及 skill、MCP、内部 RPC、schema、专用协议时，选择 `custom_tool_and_protocol_compliance`。
- 涉及多模块或 sub-agent 并行时，选择 `parallel_workflow_coordination`。
- 涉及密钥、权限、安全、隐私或破坏性操作时，选择 `security_privacy_and_secret_handling` 或 `architecture_boundaries_and_security_compliance`。

## 4. 输出边界

评分输出遵循以下边界：

1. `quality.toml`：机器可解析评分内容，只能包含选中的 7-10 个 `[[criterion]]`。
2. Bad Pattern 命中说明：只在评分说明中额外指出，不写入 `quality.toml`，也不生成单独文件。

如果当前接口只能返回一个要被 TOML parser 直接读取的文本，必须只返回纯 TOML；Bad Pattern 命中说明不能附在 TOML 前后。

评分说明中的 Bad Pattern 写法：

```text
Bad Pattern 命中：
- 只做准备动作：模型只加载 skill 和列计划，没有读取关键文件或执行补救动作。
- 错误工具/协议：任务要求内部 RPC 客户端，但模型生成了不可用的 curl 脚本。
```

如果未命中，写：

```text
Bad Pattern 命中：未发现明确命中。
```

## 5. TOML 输出格式

输出必须是合法 TOML，且只能输出 TOML 内容：

- 不得包含 Markdown 代码块、标题、解释、前后缀文本。
- 顶层使用重复表 `[[criterion]]`。
- 必须包含 7-10 个 criterion。
- criterion 的 `name` 必须来自本文 20 个候选维度。
- 每个 criterion 必须包含固定字段：`name`、`description`、`type`、`points`、`weight`、`score`、`rationale`。
- `description` 必须由评分 AI 结合本次真实 JSONL、任务类型、验收标准和该 criterion 自主生成，不得复制固定模板。
- `description` 必须是中文单行详细评分标准，包含该维度在本次任务下的 1-5 分各档位具体定义，格式大致为“该维度评价什么？1分：...。2分：...。3分：...。4分：...。5分：...。”
- `description` 不得只写一句维度摘要，不得使用英文描述、占位句、“保持模板原文不变”或与本次任务无关的泛用模板。
- 所有字符串字段都必须使用普通双引号 `"..."`，不得使用 TOML 多行字符串。
- `description` 必须写成一行 TOML 字符串，不得换行保存。
- `rationale` 也必须写成一行 TOML 字符串；如果内容较长，也不要使用 TOML 多行字符串。
- `type` 固定为 `"likert"`。
- `points` 固定为 `5`。
- `weight` 固定为 `1.0`。
- 评分 AI 应根据证据生成 `description`、`score` 和 `rationale`；`name` 必须来自候选维度池，`type`、`points`、`weight` 按本文规则生成。
- `score` 必须是整数，范围为 `1` 到 `points`。当前模板中 `points = 5`，因此分数范围为 `1..5`。
- `rationale` 必须使用中文，说明评分依据、主要证据、完成度和扣分点；不要写 Bad Pattern 标签命中结论。
- `rationale` 内如需引用命令、文件名或用户原话，优先使用反引号、中文引号或转义双引号，保证 TOML 仍可解析。
- 不应新增额外表、额外字段或遗漏模板字段。

单个 criterion 的结构如下：

```toml
[[criterion]]
name = "user_experience_and_interaction"
description = "交互过程中的用户体验是否顺畅高效？1分：工作流死锁、循环错误、没有有效输出，完全不可用。2分：推理冗长，反复自我纠错，存在明显逻辑偏差，需要用户频繁干预。3分：存在一些重复和冗余推理，偶尔过度复杂化，但用户仍能等待并获得结果。4分：整体体验良好，偶有冗余或自我修正，但不影响任务完成。5分：推理高效，逻辑清晰，响应顺畅，准确抓住核心意图，输出可信且几乎无优化空间。"
type = "likert"
points = 5
weight = 1.0
score = 3
rationale = "轨迹中能看到模型先读取相关请求链路并实际运行 `npm run lint`，说明主路径有推进和验证。但 JSONL 里没有看到针对并发刷新场景的自动化测试，且最终回复声称完全修复，与仍未覆盖的初始化清理路径不一致，因此该维度给3分。"
```

禁止输出以下形式：

- `description = "保持模板原文不变"`
- `description = "Keep the original template unchanged"`
- 只有一句摘要、没有 1-5 分档位定义的 `description`
- 复制与当前任务无关的固定模板描述，没有结合本次 JSONL 轨迹、任务类型或验收标准
- 使用 TOML 多行字符串保存任意字符串字段；所有字符串字段都必须是一行普通双引号字符串。
- 换行保存的多行 `description` 或多行 `rationale`
- 中英文混杂的 `description`
- 空 description 或与对应 criterion 不匹配的 description

## 6. rationale 写作要求

`rationale` 是人工评分理由，不是模板化摘要。生成时必须符合以下要求：

- 必须结合真实 trajectory、窗口 transcript、工具调用结果、人工备注或验收标准中的具体证据。
- 应写出该维度为什么给这个分数，至少包含一个完成度证据和一个扣分点；满分时也要说明为什么没有明显扣分点。
- 应体现人工质检用语，语气可以直接、具体、审慎，例如“主链路完成但闭环不足”“验证证据不足”“工具失败后 fallback 不充分”“没有看到对应文件修改证据”。
- 不得所有维度都使用相似句式或雷同描述；每条理由必须围绕该 criterion 的关注点独立判断。
- 不得只写“整体表现良好”“基本完成任务”“有一些问题”“符合要求”等空泛结论。
- 不得凭最终回复自述打分；如果最终自述和轨迹证据冲突，应以轨迹证据为准，并在理由中说明。
- 不得编造文件名、命令、测试结果、工具调用或用户反馈；无法从证据确认时，应写“未看到相关证据”。
- 可以引用具体命令、文件、模块、工具失败、测试结果、错误输出、未完成步骤或用户纠错，但不需要逐行复述轨迹。

不合格示例：

```toml
rationale = "模型整体完成较好，但还有一些不足，所以给3分。"
```

合格示例：

```toml
rationale = "轨迹中能看到模型修改了认证请求链路，并实际运行了 `npm run lint`，说明主路径有推进和验证。但 JSONL 里没有看到针对 refresh 竞争场景的自动化测试，且最终回复声称完全修复，与仍未覆盖的初始化清理路径不一致，因此该维度给3分。"
```

## 6.1 高分校准与封顶规则

评分应默认从 3 分基准开始，根据真实证据上调或下调。3 分表示主线有一定推进但存在明显缺口；4 分表示主链路基本闭环且只有轻微问题；5 分是稀有档，必须同时具备明确正向证据、关键路径闭环、无明显反证、无相关 Bad Pattern。

高分不是因为“没有看到错误”而给出，而是因为轨迹中能看到足够强的完成证据。给 5 分时，`rationale` 必须写出为什么该维度没有明显扣分点；如果只能写泛泛表述，最高只能给 4 分。

通用封顶规则：

- 如果某个维度缺少直接轨迹证据，最高给 3 分；如果关键证据完全缺失，通常给 2 分或更低。
- 如果任务要求代码、文件或产物变更，但轨迹中没有对应修改证据，`delivery_completeness_and_usability`、`engineering_quality_and_completeness` 最高给 2 分。
- 如果任务需要验证但没有运行测试、构建、lint、smoke test 或等价检查，`testing_and_verification_rigor` 最高给 2 分，`delivery_completeness_and_usability` 最高给 4 分。
- 如果只依赖最终回复自述，缺少工具结果、文件变更或测试证据，`evidence_grounding_and_trace_fidelity` 最高给 2 分，相关交付维度最高给 3 分。
- 如果工具失败后没有有效 fallback，`tool_usage_and_failure_recovery` 最高给 3 分；如果失败直接导致任务未完成，最高给 2 分。
- 如果用户需要多次提醒同一约束或纠正同一方向，`instruction_compliance_and_constraint_adherence`、`user_experience_and_interaction`、`task_planning_and_execution_control` 最高给 3 分。
- 如果命中 Bad Pattern，相关维度通常最高给 3 分；严重命中或直接导致任务失败时最高给 2 分。Bad Pattern 仍只在评分说明中额外指出，不写入 `quality.toml`。
- 如果最终回复夸大完成度、遗漏失败、遗漏验证限制或与轨迹证据冲突，`final_response_and_handoff_quality` 最高给 3 分，`evidence_grounding_and_trace_fidelity` 最高给 3 分。
- 如果维度描述和本节封顶规则发生冲突，以本节封顶规则为准。

## 7. 20 个候选评分维度

生成 `quality.toml` 时只能从以下 20 个候选维度中选择 7-10 个。`name` 必须严格使用表中英文标识；`description` 不从表中复制，而是由评分 AI 结合本次真实 JSONL 自主生成一行中文 1-5 分档位标准。

| name | 适用方向与证据 |
| --- | --- |
| `user_experience_and_interaction` | 用户体验、交互节奏、是否卡住、是否让用户反复介入 |
| `task_planning_and_execution_control` | 计划、todo、阶段推进、执行控制、失败后的策略调整 |
| `semantic_understanding_and_logical_reasoning` | 意图理解、代码定位、逻辑判断、隐含需求识别 |
| `instruction_compliance_and_constraint_adherence` | 用户约束、项目规则、多轮补充、格式和输出边界 |
| `engineering_quality_and_completeness` | 代码质量、实现完整性、错误处理、边界覆盖、测试意识 |
| `delivery_completeness_and_usability` | 最终产物、可运行性、可用性、显式和隐式需求闭环 |
| `architecture_boundaries_and_security_compliance` | 架构边界、安全合规、高风险操作、无关依赖和越界修改 |
| `tool_usage_and_failure_recovery` | tool_use、tool_result、is_error、stderr/stdout、fallback |
| `evidence_grounding_and_trace_fidelity` | 最终自述、证据一致性、虚假交付、是否忠实于轨迹 |
| `testing_and_verification_rigor` | 测试、构建、lint、dry-run、smoke test、断言和失败处理 |
| `context_exploration_and_code_navigation` | 文件读取、搜索、调用链定位、项目规则和入口识别 |
| `requirements_clarification_and_scope_control` | 模糊需求、多轮沟通、scope、是否任意扩大或遗漏 |
| `environment_and_dependency_handling` | 依赖、环境、版本、lockfile、解释器、平台差异 |
| `attachment_and_artifact_handling` | PDF、图片、zip、日志、xmind、二进制产物和解析 fallback |
| `external_research_and_source_use` | web search、GitHub、官方文档、release note、一手来源 |
| `custom_tool_and_protocol_compliance` | MCP、skill、RPC、内部工具、schema、协议调用 |
| `parallel_workflow_coordination` | 多模块、sub-agent、多源分析、冲突检查、统一汇总 |
| `security_privacy_and_secret_handling` | token、secret、auth、privacy、权限、脱敏和审计 |
| `maintainability_and_change_minimality` | diff 范围、抽象、依赖、命名、可维护性和元数据 churn |
| `final_response_and_handoff_quality` | 最终回复、handoff、验证说明、限制说明、剩余风险 |

## 8. Bad Pattern 扣分映射

| Bad Pattern | 额外指出的写法 | TOML 中如何体现 |
| --- | --- | --- |
| 任务偷懒 | 指出只完成表面显式需求，缺少自然必要的错误处理、状态处理、边界处理或验证。 | 可在 `engineering_quality_and_completeness`、`delivery_completeness_and_usability`、`testing_and_verification_rigor` 中扣分。 |
| 不主动澄清 | 指出业务口径明显不唯一却直接改，或用户补充后未吸收，或任务切换后继续旧目标。 | 可在 `requirements_clarification_and_scope_control`、`semantic_understanding_and_logical_reasoning`、`instruction_compliance_and_constraint_adherence` 中扣分。 |
| 该搜索不搜索 | 指出上游版本/官方规范相关问题只在本地猜测，或编造 issue/PR，或未引用真实外部证据。 | 可在 `external_research_and_source_use`、`evidence_grounding_and_trace_fidelity`、`delivery_completeness_and_usability` 中扣分。 |
| 环境误判 | 指出忽略 README、lockfile、environment、使用错误解释器、失败后无限安装或乱改依赖。 | 可在 `environment_and_dependency_handling`、`tool_usage_and_failure_recovery`、`engineering_quality_and_completeness` 中扣分。 |
| 忽略项目规则/skill | 指出未读或未遵守 `AGENTS.md`、`CLAUDE.md`、skill、TESTING_GUIDE，或只加载不执行。 | 可在 `custom_tool_and_protocol_compliance`、`instruction_compliance_and_constraint_adherence`、`task_planning_and_execution_control` 中扣分。 |
| 附件处理不足 | 指出未检查附件路径、要求用户手动转写、不尝试 OCR、解压、转换、解析或备选路径。 | 可在 `attachment_and_artifact_handling`、`user_experience_and_interaction`、`delivery_completeness_and_usability` 中扣分。 |
| 只做准备动作 | 指出只有开场、计划、todo、加载 skill，没有读取关键文件、修改、生成或验证。 | 可在 `task_planning_and_execution_control`、`delivery_completeness_and_usability`、`tool_usage_and_failure_recovery` 中扣分。 |
| 错误工具/协议 | 指出需要 MCP、RPC、内部客户端却用 shell/curl 猜测、参数不符合 schema 或结果不可执行。 | 可在 `custom_tool_and_protocol_compliance`、`tool_usage_and_failure_recovery`、`architecture_boundaries_and_security_compliance` 中扣分。 |
| 平台误判 | 指出 macOS/iOS 任务套 Linux 命令，或忽略权限、plist、entitlements、签名、打包。 | 可在 `environment_and_dependency_handling`、`semantic_understanding_and_logical_reasoning`、`delivery_completeness_and_usability` 中扣分。 |
| 并行整合不足 | 指出多模块任务无边界、sub-agent 输出未汇总、共享文件冲突或缺少统一验证。 | 可在 `parallel_workflow_coordination`、`maintainability_and_change_minimality`、`testing_and_verification_rigor` 中扣分。 |

## 9. 得分率计算

每个 criterion 有：

- `score`: 实得分。
- `points`: 该项满分，当前均为 `5`。
- `weight`: 权重，固定为 `1.0`。

得分率公式：

```text
passrate = sum(score * weight) / sum(points * weight)
```

不要硬编码分母。因为每次会从 20 个候选维度中选择 7-10 个 criterion，分母必须按实际输出的 criterion 数量、points 和 weight 动态计算。

## 10. 成对筛选规则

固定工作流会同时生成：

- `quality_qwen.toml`
- `quality_claude.toml`

生成后执行成对 gate。只有同时满足以下条件才通过：

1. `qwen passrate < 0.7`
2. `claude passrate > 0.7`
3. `claude passrate - qwen passrate > 0.2`

这里使用绝对差值，不再使用相对分差。只有 qwen 未达到基础通过线、claude 达到基础通过线，且两者 passrate 绝对差值大于 0.2 时，才说明质量差距足够明显。

当前评分范围为 1-5，因此正常情况下 `qwen passrate` 不会为 0；如果上游数据缺失导致无法计算 passrate，gate 不通过。

不通过时，评分文件会保留，但 workflow 不继续写提交包，并在生成评分步骤报失败。通过或失败结果会记录在 `runs/<run_id>/workflow.json` 的 `qualityGate` 字段中。

## 11. 校验清单

生成后的 `quality.toml` 应通过以下检查：

- TOML 可被 `tomllib.loads` 成功解析。
- 包含 7-10 个 `[[criterion]]`。
- 每个 criterion 的 `name` 都来自本文 20 个候选维度。
- 同一任务的 qwen 与 claude 使用同一份共享评分框架，criterion 顺序、`name`、`description`、`type`、`points`、`weight` 逐字一致，只有 `score` 和 `rationale` 可以不同。
- 评分过程先使用快速索引和轨迹摘要，再按选中维度定点回读 JSONL，没有默认逐行展开完整 JSONL。
- 没有遗漏任一必需字段。
- `score` 是整数且在 `1..5` 之间。
- 4 分和 5 分已经按 `高分校准与封顶规则` 复核；5 分必须有直接证据闭环，不能因为“没看到问题”就给满分。
- `description` 是评分 AI 结合本次 JSONL、任务类型、验收标准和该 criterion 自主生成的中文单行详细评分标准。
- `description` 包含该 criterion 在本次任务下的 1-5 分档位定义，且不是一句摘要、英文描述、占位句、中英混杂文本、空值或与当前任务无关的固定模板。
- 所有字符串字段都使用普通双引号，不使用 TOML 多行字符串。
- `rationale` 为中文，且能对应真实 JSONL、transcript、人工备注或工具结果。
- 各条 `rationale` 分别围绕对应 criterion 写作，不能相互雷同、套模板或只替换维度名。
- 每条 `rationale` 至少包含具体轨迹证据、完成度判断和扣分/给分依据。
- `rationale` 没有编造未在 trajectory、transcript、人工备注或工具结果中出现的证据。
- 文件不包含 Markdown 包裹或额外解释文本。
- 计算出的 passrate 与 `sum(score * weight) / sum(points * weight)` 一致。
- 成对 workflow 中，`qualityGate` 三个条件都按 qwen 与 claude 的 passrate 计算：`qwen passrate < 0.7`、`claude passrate > 0.7`、`claude passrate - qwen passrate > 0.2`。
- Bad Pattern 是否已在评分说明中额外指出，且没有写入 `quality.toml` 字段或 `rationale`，也没有生成单独文件。
- 是否把工具失败、附件处理、环境诊断、项目规则、外部搜索、并行整合等证据映射到合适维度。
- 是否避免仅凭模型最终自述给高分。
