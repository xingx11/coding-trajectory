# Coding Trajectory 数据提取新手教程

本文面向第一次接触命令行、Python、Git 和 Claude Code CLI 的同学，目标是从一台干净的 Windows 电脑开始，一步步完成 Coding Trajectory 数据提取流程。

本文以 Windows 10/11 + PowerShell 为例。所有命令都在项目根目录 `D:\A3Code\Coding Trajectory` 中执行，除非特别说明。

---

## 0. 先理解我们要做什么

这套流程不是普通的网页或软件项目，而是一个“数据生产流水线”。它会让 Qwen 和 Claude 分别完成同一个真实代码任务，然后把全过程记录下来，最后生成可以提交的数据文件。

一条完整数据大致包含：

| 文件 | 作用 |
|---|---|
| `trajectory JSONL` | 模型完成任务的完整对话和工具调用记录 |
| `score TOML` | 对模型完成质量的评分文件 |
| `metadata` | 任务背景和补充说明 |
| `submission.csv` | 最终提交用的汇总表 |

整个流程可以理解成：

```text
安装工具 -> 配置密钥和路径 -> 生成或准备任务 -> 运行模型 -> 收集 trajectory -> 评分 -> 汇总 -> 校验
```

## 2. 安装 Python

### 2.1 下载 Python

1. 打开 [Python 官网下载页](https://www.python.org/downloads/)。
2. 下载 Windows 版本安装包。
3. 双击安装包。
4. 在安装界面最下方勾选 `Add python.exe to PATH`。
5. 点击 `Install Now`。

### 2.2 检查 Python 是否安装成功

重新打开一个 PowerShell，输入：

```powershell
python --version
```

如果看到类似下面的输出，说明安装成功：

```text
Python 3.11.x
```

如果 `python --version` 不可用，也可以试：

```powershell
py --version
```

### 2.3 检查 pip 是否可用

`pip` 是 Python 的依赖安装工具。输入：

```powershell
python -m pip --version
```

如果能看到版本号，说明 pip 可用。

### 2.4 安装推荐的 Python 依赖

当前 pipeline 主要使用 Python 标准库，但项目搜索阶段会优先使用 `requests` 访问 GitHub/Gitee API；如果没有安装 `requests`，代码会自动退回到 `curl`。为了让新手环境更稳定，建议安装 `requests`：

```powershell
python -m pip install requests
```

如果以后要运行测试，可能需要安装 `pytest`：

```powershell
python -m pip install pytest
```

## 3. 安装 Git

### 3.1 下载 Git

1. 打开 [Git for Windows 下载页](https://git-scm.com/download/win)。
2. 下载并运行安装包。
3. 安装过程中如果不确定怎么选，保持默认选项即可。
4. 看到类似 `Git from the command line and also from 3rd-party software` 的选项时，保持默认或选择该项。

### 3.2 检查 Git 是否安装成功

重新打开 PowerShell，输入：

```powershell
git --version
```

如果看到类似下面的输出，说明安装成功：

```text
git version 2.xx.x.windows.x
```

### 3.3 中国大陆网络环境下配置代理

如果你访问 GitHub 或 `git clone` 经常失败，需要配置代理。下面的端口 `7897` 只是示例，请改成你自己代理软件里的 HTTP 端口。

```powershell
git config --global http.proxy http://127.0.0.1:7897
git config --global https.proxy http://127.0.0.1:7897
git config --global http.sslBackend schannel
```

检查是否配置成功：

```powershell
git config --global --get http.proxy
git config --global --get https.proxy
git config --global --get http.sslBackend
```

如果不需要代理，可以跳过这一节。

### 3.4 代理软件示例：Clash Verge Rev

建议准备一个本地代理软件。这里以 Clash Verge Rev 为例。

Clash Verge Rev 是一个带图形界面的代理客户端，适合新手使用。它的官方安装包和文档都在项目自己的 Release 和文档页里，不要去下来路不明的第三方包。

官方入口：

- [Release 页面](https://github.com/clash-verge-rev/clash-verge-rev/releases)
- [项目文档](https://clash-verge-rev.github.io)

最简单的使用步骤：

1. 安装并打开 Clash Verge Rev。
2. 导入你的订阅链接、配置文件或节点配置。
3. 选择一个可用节点。
4. 打开 `System Proxy`。
5. 一般保持 `Rule` 模式就够用；如果你不熟悉规则，先按服务提供方的默认建议来。
6. 查看本地代理端口，常见是 `127.0.0.1:7897`，但以你客户端里实际显示的端口为准。

这一步做完以后，浏览器通常已经能走代理了。但这个项目里还要再单独配两处：

- `git config --global http.proxy ...`
- `.env` 里的 `HTTP_PROXY=...`

原因是：

- Clash Verge Rev 的 `System Proxy` 主要影响浏览器和会读取系统代理的程序。
- 这个项目里，Git 和 Python 的网络请求并不一定自动跟着系统代理走。
- 所以要把代理地址显式写给 Git 和 `.env`，这样更稳。

如果你的 Clash Verge Rev 端口不是 `7897`，请把下面所有命令和 `.env` 里的端口一起改成你实际的端口。

[仪表盘 - 良心云](https://xn--9kqz23b19z.com/#/dashboard)

## 7. 配置项目环境变量

### 7.1 复制 `.env` 文件

在项目根目录执行：

```powershell
copy .env.template .env
```

`.env.template` 是模板，`.env` 才是真正使用的配置文件。

### 7.2 编辑 `.env`

用记事本打开：

```powershell
notepad .env
```

需要填写这些内容：

```env
QWEN_AUTH_TOKEN=<你的 Qwen token>
QWEN_BASE_URL=<你的 Qwen API 地址>
QWEN_MODEL=qwen3.7-max

CLAUDE_AUTH_TOKEN=<你的 Claude token>
CLAUDE_BASE_URL=<你的 Claude API 地址>
CLAUDE_MODEL=claude-opus-4-6-20260205

GITHUB_TOKEN=<可选，但推荐填写>
GITEE_TOKEN=<使用 Gitee 搜索时需要>
HTTP_PROXY=<可选，中国大陆访问 GitHub 时通常需要>
```

注意：

- 不要把 `.env` 发给别人。
- 不要把 token 写到聊天窗口、文档或代码里。
- `HTTP_PROXY` 示例是 `http://127.0.0.1:7897`，端口要按你自己的代理软件填写。
- 如果使用 `gen --source gitee`，通常不需要 `HTTP_PROXY`，但需要 `GITEE_TOKEN`。

## 8. 配置 `tasks.toml`

`tasks.toml` 是任务配置文件。先用记事本打开：

```powershell
notepad tasks.toml
```

最上方有一段 `[batch]`，重点改这几个值：

```toml
[batch]
delivery_date = "20260605"
runs_root = "D:\\A3Code\\runs"
max_parallel = 3
```

字段解释：

| 字段 | 含义 |
|---|---|
| `delivery_date` | 本次交付批次日期，会生成 `delivery_YYYYMMDD` 目录 |
| `runs_root` | 模型实际运行任务时用的临时代码目录 |
| `max_parallel` | 同时跑多少个 `claude -p` 进程 |

建议初学者先把 `max_parallel` 设置为 `3`，确认流程跑通后再提高。

如果 `runs_root` 目录不存在，可以手动创建：

```powershell
New-Item -ItemType Directory -Force "D:\A3Code\runs"
```

## 10. 生成任务

任务可以手动写，也可以让 `gen` 自动生成。新手建议先用自动生成。

### 10.1 使用 Gitee 生成任务

如果在国内网络环境下，优先试 Gitee：

```powershell
python -m ctpipe gen --count 3 --per-project 3 --source gitee
```

这条命令会尝试生成 3 条任务，并把任务追加到 `tasks.toml`，同时生成对应评分模板。

### 10.2 使用 GitHub 生成任务

如果 GitHub 访问稳定，可以用：

```powershell
python -m ctpipe gen --count 3 --per-project 3
```

如果 GitHub 搜索失败，请检查：

- `.env` 里的 `GITHUB_TOKEN` 是否填写。
- `.env` 里的 `HTTP_PROXY` 是否正确。
- Git 代理是否配置。

### 10.3 只预览，不真正写入

如果只是想测试参数，可以加 `--dry-run`：

```powershell
python -m ctpipe gen --count 3 --per-project 3 --dry-run
```

## 11. 运行完整 pipeline

确认 `tasks.toml` 里已经有 `[[task]]` 任务后，执行：

```powershell
python -m ctpipe --config tasks.toml --env .env all
```

这条命令会依次运行 6 个阶段：

| 阶段 | 命令 | 做什么 |
|---|---|---|
| 1 | `prepare` | 准备运行目录和交付目录 |
| 2 | `run` | 让 Qwen 和 Claude 执行编码任务 |
| 3 | `collect` | 收集 Claude Code 生成的 trajectory JSONL |
| 4 | `score` | 自动生成初始评分 |
| 5 | `finalize` | 计算 passrate 并生成 `submission.csv` |
| 6 | `validate` | 校验文件是否完整 |

如果想分步执行，也可以这样：

```powershell
python -m ctpipe prepare
python -m ctpipe run
python -m ctpipe collect
python -m ctpipe score
python -m ctpipe finalize
python -m ctpipe validate
```

## 12. 只运行指定任务

如果只想跑一个任务，例如 `CT-0001`：

```powershell
python -m ctpipe prepare --tasks CT-0001
```


```powershell
python -m ctpipe run --tasks CT-0001 --models qwen
```

## 14. 人工复核评分

自动评分只是初稿，交付前需要人工检查。

打开评分文件：

```powershell
notepad delivery_20260605\scores\qwen\CT-0001.quality.toml
notepad delivery_20260605\scores\claude\CT-0001.quality.toml
```

每个评分文件一般有 7 个维度，每项 0 到 5 分。改完以后重新执行：

```powershell
python -m ctpipe finalize
python -m ctpipe validate
```

最终 passrate 规则：

```text
qwen passrate  < 0.7
claude passrate >= 0.71
claude passrate > qwen passrate
```

如果不满足规则，需要检查任务是否合格、评分是否合理，必要时重跑或替换任务。

## 15. 断点续跑和重跑

pipeline 会把状态写入：

```text
delivery_20260605/pipeline_state.json
```

如果中途失败，通常直接重新运行即可：

```powershell
python -m ctpipe --config tasks.toml --env .env all
```

已经成功的阶段会自动跳过。

如果你想强制重跑某个任务的某些阶段，例如重跑 `CT-0001` 的 `run` 和 `collect`：

```powershell
python -m ctpipe reset --tasks CT-0001 --stages run collect
python -m ctpipe run --tasks CT-0001
python -m ctpipe collect --tasks CT-0001
```

## 16. 常见问题

### 16.1 `python` 不是内部或外部命令

原因通常是安装 Python 时没有勾选 `Add python.exe to PATH`。

处理方式：

1. 重新运行 Python 安装包。
2. 选择 `Modify`。
3. 勾选 PATH 相关选项。
4. 安装完成后重新打开 PowerShell。

也可以试：

```powershell
py --version
```

### 16.2 `claude` 不是内部或外部命令

说明 Claude Code CLI 没有安装成功，或安装目录没有加入 PATH。

处理方式：

1. 重新打开 PowerShell。
2. 执行 `claude --version`。
3. 如果仍失败，回到 [Claude Code Installation](https://code.claude.com/docs/en/installation) 按官方文档重装。

### 16.3 GitHub clone 很慢或失败

先检查 Clash Verge Rev 是否已经：

- 已启动
- 已导入配置
- 已选择可用节点
- 已打开 `System Proxy`

然后再检查 Git 代理：

```powershell
git config --global --get http.proxy
git config --global --get https.proxy
```

如果代理端口不对，重新设置：

```powershell
git config --global http.proxy http://127.0.0.1:7897
git config --global https.proxy http://127.0.0.1:7897
```

如果仍然失败，可以改用 Gitee：

```powershell
python -m ctpipe gen --count 3 --per-project 3 --source gitee
```

### 16.4 `.env` 配了代理，但 Git 还是失败

`.env` 里的 `HTTP_PROXY` 是给 Python 的 GitHub API 搜索用的，Git 自己还需要 `git config --global http.proxy ...`。

另外，Clash Verge Rev 只开了 `System Proxy` 还不够稳，最好把 Git 和 `.env` 都显式指向本地代理端口。

两边都要配：

```powershell
git config --global http.proxy http://127.0.0.1:7897
git config --global https.proxy http://127.0.0.1:7897
```

并在 `.env` 里写：

```env
HTTP_PROXY=http://127.0.0.1:7897
```

### 16.5 `validate` 报文件缺失

常见原因：

- `run` 阶段没有成功完成。
- `collect` 没有找到对应的 JSONL。
- `score` 没有生成评分文件。
- `metadata` 文件没有补齐。

处理方式是先看 `pipeline_state.json`，确认失败阶段，然后从失败阶段重新跑。

### 16.6 不小心把任务或评分改坏了

如果只是评分文件改错，可以从 `rubrics_templates/` 重新复制模板，或让 pipeline 的 `score` 阶段重新生成。

如果某个阶段状态卡住，可以 reset：

```powershell
python -m ctpipe reset --tasks CT-0001 --stages score
python -m ctpipe score --tasks CT-0001
python -m ctpipe finalize --tasks CT-0001
python -m ctpipe validate --tasks CT-0001
```

## 17. 新手推荐的最小跑通流程

第一次不要直接跑 45 条任务，先跑 1 到 3 条确认环境没问题。

```powershell
cd "D:\A3Code\Coding Trajectory"

python --version
git --version
claude --version

copy .env.template .env
notepad .env
notepad tasks.toml

python -m ctpipe gen --count 3 --per-project 3 --source gitee
python -m ctpipe --config tasks.toml --env .env all

python -m ctpipe finalize
python -m ctpipe validate
```

如果这套流程能跑通，再把 `--count 3` 改成正式需要的数量，例如：

```powershell
python -m ctpipe gen --count 45 --per-project 3 --source gitee
python -m ctpipe --config tasks.toml --env .env all
```

## 18. 交付前检查清单

交付前逐项确认：

- `python -m ctpipe validate` 通过。
- `delivery_YYYYMMDD/submission.csv` 已生成。
- 每个任务都有 Qwen 和 Claude 的 trajectory。
- 每个任务都有 Qwen 和 Claude 的评分文件。
- 每个任务都有 metadata 文件。
- `.env` 没有被提交或发送给别人。
- passrate 满足 `qwen < 0.7`、`claude >= 0.71`、`claude > qwen`。
