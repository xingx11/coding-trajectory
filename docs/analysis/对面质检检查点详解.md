# 对面质检检查点详解

> 本文档基于 16 份真实质检报告（内部样本，未随仓库分发）逆向归纳，**单纯罗列验收方（"对面"）对每条 Coding Trajectory 数据所做的全部检查点**：每项的含义、通过标准，以及报告中出现过的真实失败案例。
>
> 来源样本：`CT-0011`、`D-012`、`D-024`、`D-030`、`0033(useMemo)`、`flask_auth`、`security-compliance-03`、`报告6(DeskMate)`、`bug-fix-08`、`41-609-bug-fix-31(wisdom-advisor)`，以及 0610 批"异题被剔除"四例 `3-09-0610-bug-fix-003`、`4-09-0610-bug-fix-004`、`5-09-0610-bug-fix-005`、`6-09-0610-testing-006`。

---

## 1. 总体结构

对面的质检分为两大类 grader，最终汇总为一个**通过 / 不通过**结论：

| 类别 | 中文 | 判定方式 | 覆盖检查 |
|---|---|---|---|
| **Code-based Graders** | 基础检查 | 客观，可程序化核验 | 文件完整性、Rubrics 格式、分数关系 |
| **Model-based Graders** | 质量审核 | 主观，需理解轨迹/代码 | Query 质量、Rubrics 质量、轨迹对比、分数归因 |

报告正文一般按 **5 个可见步骤**呈现，外加一层基础检查（在 D-024 / D-030 中作为末尾"基础检查结果汇总"表出现，在 bug-fix-31 等报告中隐含在格式 / 分数描述里）：

```
[基础检查]  文件完整性 → Rubrics 格式 → 分数检查        (Code-based)
步骤一      真实 Query 提取与质量审核                   (Model-based)
步骤二      Rubrics 质量评估                            (Model-based)
步骤三      轨迹对比分析                                (Model-based)
步骤四      分数差异归因                                (Model-based)
步骤五      评分对照表（passrate 复核）                  (Code-based 复算)
[结论]      质检通过 / 不通过
```

> **判定逻辑**：基础检查全部通过、且全部质量审核均达标，才判"通过"；任一 Model-based 项不达标即判"不通过"。D-024、D-030 两例文件、格式、分数关系全部合格，仅因 **Rubrics 质量（量身定制 / 事实性）不达标**就被判**不通过**——可见**通过 passrate 门槛是必要非充分条件**。同理，0610 批 `bug-fix-003`～`testing-006` 一组样本**分数门槛（含 (claude−qwen)/qwen ≥ 20%）数值上完全通过**，却因**步骤三同题公平性不达标——两模型实际不是同一道题**被整组剔除（详见 §4 步骤三、§6 模式 8）；再次印证门槛只是必要非充分条件。

## 3. 基础检查（Code-based）逐项详解

### A1 上传文件完整性
- **含义**：一条数据必须包含 4 个文件——qwen 轨迹 jsonl、claude(opus) 轨迹 jsonl、qwen 评分 toml、claude 评分 toml。
- **通过标准**：4 文件齐全。
- **真实示例**：D-024 列出 `D-024.jsonl / D-024_opus.jsonl / D-024.quality.toml / D-024_opus.quality.toml`；D-030 列出 `D-030.jsonl / D-030 (1).jsonl / D-030.quality.toml / D-030.quality (1).toml`（命名不统一，但齐全即可）。

### A2 Rubrics 文件格式
- **含义**：两份 TOML 结构合法、字段齐全。
- **通过标准**：
  - 可被解析（`tomllib`）；
  - 顶层为 `[[criterion]]` 数组；
  - 每条 criterion 含 `name / description / type / score / weight / points / rationale` 七个字段；
  - `type == "likert"`；`points == 5`。

### A3 分数检查
- **含义**：passrate 关系满足验收门槛，且打分形态合法。
- **通过标准（逐子项）**：
  - **A3-a** qwen passrate **<** claude passrate；
  - **A3-b** `(claude − qwen) / qwen ≥ 20%`；
  - **A3-c** qwen passrate **< 0.7**；
  - **A3-d** 所有 score ∈ {1,2,3,4,5}（整数，无 0）；
  - **A3-e** 每条 description 含 1–5 分档梯度文字；
  - **A3-f** 至少存在 1 个 qwen==claude 的持平维度（多份报告将"有持平项"作为标注未刻意拉开差距的佐证）。
- **passrate 公式**：`Σ(score_i × weight_i) / Σ(points_i × weight_i)`。
- **真实示例**：D-030 得分率 60.0% vs 80.0%，差距**恰好 20.0%**——属"刚好达标"，说明 20% 是硬下限。

## 4. 质量审核（Model-based）逐项详解

### 步骤一 · Query 质量

| 子项 | 含义 | 通过标准 | 失败信号 |
|---|---|---|---|
| **原创性 B1-a** | 不是抄公开评测集 | 真实业务 / 内部工具场景 | 与 HumanEval / SWE-bench 等雷同 |
| **真实性 B1-b** | 像真人求助 | 口语化、贴近真实开发 / 运维 | 命题作文式、不像真实用户 |
| **信息充分性 B1-c** | 既明确又有探索空间 | 含目标 + 场景 + 现象，但不直接给答案 | 信息过少无法定位 / 过多无探索空间 |
| **一致性 B1-d** | 两模型同题 | 首轮 Query 逐字一致 | 两侧 Query 不同（难度不可比） |

> 示例：bug-fix-31 的 Query"我用 threadsaffinity 追踪一个进程……是不是哪里没清理干净？"被判三项全过——口语化、含工具名 + 触发场景 + 推测方向、留有跨模块探索空间。后续轮次（"需要"/"好了吗"）也基本一致。

### 步骤二 · Rubrics 质量（**报告中最高频的不通过点**）

| 子项 | 含义 | 通过标准 | 真实失败案例 |
|---|---|---|---|
| **原子性 B2-a** | 每个维度只评一个评估点 | description 无"且 / 同时 / 并"叠加多要求 | D-012 因原子性不通过 |
| **量身定制 B2-b** | 紧扣本任务 | description 含具体文件名 / 函数名 / 技术点 | **D-024、D-030**：`tool_invocation_efficiency`、`trajectory_length`、`environment_platform_awareness` 三维完全未提及本任务（pytest 断言 / owner 属性），属"万能模板"→ 不通过 |
| **唯一性 B2-c** | 维度间不重叠 | 不会对同一事件重复扣分 | **CT-0011**：`tool_usage` 与 `task_planning` 都引用"12 次错误 / 337 行"对 qwen 重复扣分；**bug-fix-31**：`context_exploration` 与 `tool_usage` 都因"路径错误"扣分 |
| **事实性 B2-d** | rationale 有据 | 引用行号 / is_error 次数 / 文件名 / 函数名 / 测试结果 | **D-024、D-030**：rationale 充斥"基本正确""基本充分"无具体证据；D-030 更出现自相矛盾——`code_modification` 先写"没有任何文件修改"又写"修改基本正确但可能遗漏" |
| **评分梯度** | 规则明确 | description 给 1–5 分档（与 A3-e 重合） | — |
| **Explicit/Implicit 分类 B2-e** | 维度类型标注 | 每维标 Explicit（可直接检查"做了什么"）或 Implicit（推断"做得多好"）并统计分布 | 各报告均给出，如 bug-fix-31 = Explicit 4 / Implicit 4 |

> **Explicit / Implicit 判别要点**：
> - *Explicit*：可对照交付物 / 轨迹直接核验，如"是否实际修改了文件""is_error 次数"。
> - *Implicit*：需从过程推断质量，如"根因理解深度""执行规划是否高效"。

### 步骤三 · 轨迹对比分析

| 子项 | 含义 | 通过标准 | 失败信号 |
|---|---|---|---|
| **Query 一致性** | 同题 | 逐字一致（与 B1-d 同源） | 两侧题目不同；**最严重为两模型直接异题**（见下方反例） |
| **环境一致性 B3-a** | 同环境 | 同项目副本、同 OS、同 cwd | 一方环境更优 |
| **难度公平性 B3-b** | 难度对等 | 无一方更易、无作弊、无跳验证 | 一方被喂答案 / 任务被简化 / **直接换题（异题）** |
| **交互方式一致 B3-c** | 引导对等 | 轮次相当、无单方额外提示 | 一方获得额外 hint |

报告同时给出**基础统计对比表**（总消息数、Assistant 轮次、工具调用总数、Read/Edit/Write 次数、新增 / 修改文件数）、**架构方案对比**、**关键缺陷**、**独特优势**，作为归因与评分的事实底座。

> 公平性结论可以是"公平但 qwen 自身失误"——如 bug-fix-31：qwen 工具调用反而更多（50 vs 30），失分源于**路径误判**（Linux 风格路径访问 Windows 工作区）造成的效率损耗，而非题目不公。

> **反例（不公平 → 轨迹对比不通过 / 整组剔除）**：0610 批 `bug-fix-003`～`testing-006` 一组样本，数据流水线给两模型派发了**完全不同的 Bug / 任务**——
> - **003**：qwen 修"文章编辑提交后内容不更新"（`editor-page.tsx:69`，并额外发现"关注作者关注错人"`article-page.tsx:75`）；claude 修"评论 Dec 29th 占位文字 + 表单 `if(name&&value)` + DOMPurify XSS"。同一项目里**毫不相关的两组缺陷**。
> - **006**：qwen 单题"取消收藏白屏"；claude 四题"测试 + 表单校验"，连**任务类别**（bug 修复 vs 测试+特性）都不同。
>
> 两侧首轮 Query **互不相同**，**同题公平性直接不通过**，head-to-head 对比与 20% 分差门槛**从根本失效**——即便数值上 `(claude−qwen)/qwen ≥ 20%`（003 实为 50%）也属**伪分差**，一律整组退回 / 重采（须让两模型针对**同一条 Query** 重跑，保证字节级同题）。此类异题样本几乎必然叠加另一处致命伤：**Rubric 仅按一侧（多为 claude）任务编写、原样套用到另一侧 → 另一侧 rationale 系统性造假**（事实性 fail，003 的 qwen 7 条 rationale 全部描述它从未做过的"占位/表单/XSS"），且 **qwen 真正完成的工作从未被任何维度评价**。

## 5. 综合与结论（步骤四 / 五 / 结论）

### C1 分数差异归因
- **含义**：把 qwen ↔ claude 的得分率差，拆成三类成因并给百分比。
- **三因素**：
  - **模型能力**（最常见主因，多在 60–70%）——如 qwen 工具错误率高、环境感知差、交付不闭环；
  - **Prompt 因素**（通常 5–10%）——同题则影响极小；
  - **标注因素**（10–60%）——重复扣分、区分度不足、模式化打分等。
- **要求**：三者合计 100%，且与步骤三事实一致。
- **异题特例**：若步骤三判定两模型**不是同一题**，归因须额外引入**数据 / 题目设计因素**（可达 ~50%，如 003）、Prompt 因素趋近 0%；但此类样本本身已因同题公平性 fail 判**不通过**，归因仅用于说明该分差为伪、不可用于能力评估。

### C2 评分对照表复核
- **含义**：逐维列 qwen / claude 分数、权重、差值，给加权总分、得分率、差距，并复算门槛。
- **通过标准**：表内数字自洽，且 passrate 与 TOML 实算一致。

### C3 质检结论
- **含义**：综合上述全部检查给出**通过 / 不通过**。
- **规则**：基础检查 + 全部质量审核均达标方为"通过"；任一 Model-based 项不达标即"不通过"（D-024 / D-030 即例）。

## 6. 报告中反复出现的"不通过 / 风险"模式

对面在真实样本里实际判罚过的典型问题（均有报告佐证）：

1. **通用模板维度（量身定制 fail）**——`tool_invocation_efficiency` / `trajectory_length` / `environment_platform_awareness` 等描述不含任务专有名词，换题可复用。（D-024、D-030）
2. **模糊 rationale（事实性 fail）**——"基本正确""基本充分""基本简洁"无行号 / 函数名 / 测试结果。（D-024、D-030）
3. **自相矛盾 rationale**——同一条先说"没有任何文件修改"又说"修改基本正确"。（D-030）
4. **重复扣分（唯一性 fail）**——两个维度对同一事件（如路径错误、工具错误次数）各扣一次。（CT-0011、bug-fix-31）
5. **模式化打分**——qwen 全 3 分 / claude 全 4 分，区分度不足。（D-030 的 qwen 全 3；CT-0011、bug-fix-31 的 claude 全 4）
6. **"修复已存在"特殊场景评分不一致**——两侧都未改代码、都确认已修复，却给 qwen 3 分、claude 4 分。（D-030）
7. **5 分标准过高**——5 分要求"内核级分析 / 运行测试验证"等近乎不可达，致 4 分成为实际上限、压缩区分空间。（bug-fix-31）
8. **两模型异题（同题公平性 fail → 轨迹对比不通过）**——数据流水线给 qwen / claude 派发了**不同的 Bug 或任务**，两侧首轮 Query 互不相同，head-to-head 对比与 20% 分差门槛从根本失效；**即便数值过 20% 也是伪分差**。几乎必然叠加"Rubric 仅按一侧编写、另一侧 rationale 整份虚构（事实性 fail）"，且被换题一侧真正完成的工作无任何维度评价。处置：**整组退回 / 字节级同题重采**，无法局部修订挽救。（`3-09-0610-bug-fix-003`、`4-09-0610-bug-fix-004`、`5-09-0610-bug-fix-005`、`6-09-0610-testing-006`）
